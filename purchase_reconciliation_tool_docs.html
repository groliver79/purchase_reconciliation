📋 About This App

The Purchase Reconciliation Tool is a Microsoft Power Apps Model-Driven application built on Dataverse. It gives Purchase Cardholders and Approving Officials a unified view of PRISM TPC (Commitment) and PCA (Obligation) data down to the allocation level, alongside US Bank purchase card transaction data. Users can document purchase request items against a TPC from various vendors and verify that purchase totals reconcile against both the TPC commitment and the PCA obligation.

User Roles

RoleResponsibilities
Purchase Cardholder Creates purchase requests, adds line items (vendor, items, recipient, estimated and actual costs), adds TPC allocations, and associates US Bank transactions to allocations.
Approving Official (Manager) Reviews purchase requests and validates reconciliation totals against TPC commitments and PCA obligations.
Audience This document is intended for developers and system administrators responsible for maintaining, extending, or deploying this solution. End-user guidance is covered in the separate User Guide.

🏗 Architecture

The solution follows a standard Power Platform layered pattern. Dataverse provides the data platform, C# server-side plugins enforce data integrity and automate record management, and JavaScript web resources provide guided user input and real-time validation on UCI model-driven forms. 

Dataverse
  ├─ biteam_purchaserequest        (Purchase Request — primary entity)
  │    ├─ Plugin: AddTpcPrefixPlugin          (Pre-Create, stage 20)
  │    └─ JS:     tpcRegexPrompt.js           (onLoad, onSave, onChange)
  │
  ├─ cr6c5_pca                     (PCA / Obligation)
  │    ├─ Plugin: AddPcaPrefixPlugin          (Pre-Create, stage 20)
  │    └─ JS:     pcaRegExPrompt.js           (onLoad, onSave, onChange)
  │
  ├─ biteam_tpcallocations         (TPC Allocation)
  │    ├─ Plugin: OccSummaryOnAllocationDeletePlugin  (Pre-Delete stage 20 · Post-Update stage 40)
  │    └─ JS:     tpcAllocation_Prepop.js     (Quick Create onLoad)
  │
  ├─ cr6c5_usbanktransactions      (US Bank Transactions)
  │    └─ Plugin: SyncOccSummaryPlugin        (Post Create/Update/Delete, stage 40)
  │
  └─ biteam_occsummary             (OCC Summary — plugin-maintained child entity)
       └─ Surfaced as read-only subgrid on Purchase Request form

⚙️ C# Plugins

All plugins share the namespace ProcurementPlugins and are compiled into a single assembly: ProcurementPlugins.dll.

AddTpcPrefixPlugin

Plugin Pre-Operation biteam_purchaserequest

Automatically prepends TPC- to the biteam_name field before a new Purchase Request record is created. Because this runs at Pre-Operation (stage 20), mutating the Target entity is sufficient — no separate service call is needed. Includes a duplicate-prefix guard to prevent stacking if the record is re-processed.

AddPcaPrefixPlugin

SyncOccSummaryPlugin

Plugin Post-Operation cr6c5_usbanktransactions

Maintains the biteam_occsummary child entity whenever a US Bank transaction is created, updated, or deleted. Fires on all three messages at Post-Operation stage 40. On Update, filtered attributes limit execution to changes in transaction amount, associated OCC, associated purchase request, associated TPC allocation, or record state — preventing unnecessary firing. When the TPC Allocation lookup changes, the plugin re-resolves the OCC from the new allocation and recalculates the old OCC summary so its total no longer includes the moved transaction's amount.

→
STEP 2bIf OCC changed: recalculate old OCC Summary total
→
STEP 3Re-aggregate all active transaction amounts for PR + new OCC
→
STEP 4Upsert OCC Summary row (update or create)

Recursion Guard This plugin writes biteam_associatedocc back to the transaction record when resolving OCC via TPC Allocation. The Depth > 1 guard is critical — it prevents the plugin from recursively triggering itself on that write-back update.
MergeWithPostImage Helper On Update, partial payloads may only contain changed fields. The MergeWithPostImage helper merges the Update Target with the PostImage so unchanged lookup fields (like Purchase Request) are always available regardless of what triggered the update.

OccSummaryOnAllocationDeletePlugin

Plugin Pre-Operation Post-Operation biteam_tpcallocations

Recalculates or removes the biteam_occsummary record when a TPC Allocation is hard-deleted or deactivated/reactivated. Registered on two steps:

Delete step runs at Pre-Operation (stage 20) so the record still physically exists in Dataverse — the deleting record is explicitly excluded from the remaining-allocations query using a NotEqual condition on its ID.

Update step runs at Post-Operation (stage 40), filtered on statecode only. When an allocation is deactivated, the statecode = 0 (active) filter on the remaining-allocations query naturally excludes it; when reactivated, it is naturally included. No manual ID exclusion is needed for this path.

Execution Flow

STEP 1Read OCC and parent PR from PreImage (Delete) or PostImage (Update)
→
STEP 2Query remaining active allocations for same OCC (exclude deleting record on Delete only)
→
STEP 3Find existing OCC Summary record
→
STEP 4Delete summary if no active allocations remain · Recalculate and update if allocations remain

Why Pre-Operation for Delete? The PreImage on a Delete step is the only reliable source of data for the record being deleted. Running at Pre-Operation (stage 20) also means the record still exists in Dataverse during execution, which allows the remaining-allocations query to run correctly with the explicit NotEqual exclusion of the deleting record's ID. The Update step uses Post-Operation (stage 40) because the statecode change is already committed, allowing the active-only filter to work without manual exclusion.
Adding More Plugins Additional plugin classes can be added to the same DLL without re-registering the assembly. Only a new Step registration is needed in the Plugin Registration Tool. If the plugin count grows significantly, consider refactoring to a single configurable plugin using the Unsecure Configuration field.

📜 JavaScript Web Resources

All JavaScript files share the ProcurementForms namespace, using the revealing module pattern (IIFE) to keep functions private and expose only the public API needed by form event wiring.

tpcRegexPrompt.js

JavaScript biteam_purchaserequest

Validates the TPC number field (biteam_name) against the required format YY-XXXXXX (two fiscal year digits, a dash, six digits). Provides real-time progressive guidance as the user types, and blocks save if the format is invalid. Only fires on Create forms (formType === 1).

Form Event Wiring

EventFunctionPass Execution Context
OnLoadProcurementForms.Tpc.onLoadYes
OnSaveProcurementForms.Tpc.onSaveYes
OnChange (biteam_name)Registered programmatically via attribute.addOnChange in onLoadN/A

Validation Stages (onChange)

CasePatternBehaviour
Empty—Restores the format hint notification
Valid/^\d{2}-\d{6}$/Clears all notifications — no action needed
Partial — year only/^\d{1,2}$/Guides user to add a dash
Partial — year + dash/^\d{2}-$/Guides user to enter 6 digits
Partial — incomplete number/^\d{2}-\d{1,5}$/Tells user exactly how many digits remain
Wrong formatNone of the aboveShows error — correct format with example

onSave Behaviour

Calls preventDefault() to block save if the value is not valid. Exception: if the value already starts with TPC- and is 13 characters long, save is allowed — this handles the case where the plugin has already run (e.g. on a re-save).

Fiscal Year Detection Current and prior fiscal year digits are derived dynamically at runtime using new Date().getFullYear(), so the hints stay accurate across fiscal years without code changes.

OData Lookup Resolution Lookup field values are retrieved using OData navigation property names (e.g. _biteam_projectcode_value) which return both the GUID and the @OData.Community.Display.V1.FormattedValue for the display name. The entity type constants in the file must match the lookuplogicalname returned in the OData response — not the field logical name on the allocation entity.
Fails Gracefully If the WebAPI query fails or no prior allocation exists, the form simply opens blank. The user can still fill in all fields manually. No error is surfaced to the user — failures are logged to the browser console only.

🚀 Deployment

Plugin Assembly

  1. Build the solution in Visual Studio in Release mode (not Debug).
  2. Open the Plugin Registration Tool and connect to the target environment.
  3. If the assembly is already registered — click Update to upload the new DLL.
  4. If registering for the first time — click Register New Assembly.
  5. Register or verify Steps for each plugin (see registration tables above).
  6. Confirm PreImage is registered on the Delete step of OccSummaryOnAllocationDeletePlugin.
  7. Confirm PostImage is registered on the Update step of OccSummaryOnAllocationDeletePlugin with statecode as the only filtering attribute.
  8. Confirm PostImage is registered on Create and Update steps of SyncOccSummaryPlugin.
  9. Test end-to-end: create a Purchase Request, confirm TPC- prefix and linked PCA record appear.

Full Solution Deployment

  1. Export the solution as Managed from the source environment via make.powerapps.com.
  2. Import into the target environment via the Power Platform Admin Center or PAC CLI.
  3. Verify plugin assembly steps are active post-import.
  4. Run an end-to-end test: create a Purchase Request → add allocations → associate a US Bank transaction → validate the OCC Summary subgrid updates correctly.
Import Failures Import failures are almost always caused by a Debug build DLL being included in the solution export. Always build in Release mode before exporting. Verify in Visual Studio: Build → Configuration Manager → confirm Active Solution Configuration is Release.
